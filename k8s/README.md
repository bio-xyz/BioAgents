# BioAgents Worker — Kubernetes manifests

Kustomize layout for deploying BullMQ workers to EKS (`overlays/eks-staging`, `overlays/eks-prod`) and kind (`overlays/kind`).

Design spec: [`../documentation/docs/K8S_WORKER_MIGRATION.md`](../documentation/docs/K8S_WORKER_MIGRATION.md).

## Layout

```
base/
  worker-deep-research/      Deployment + PDB + KEDA ScaledObject
  worker-chat/               Deployment + PDB + KEDA ScaledObject
  worker-heavy/              Deployment + PDB + KEDA ScaledObject (multi-trigger)
  worker-scheduled/          Deployment + PDB (no scaling — static 1 replica)
  keda/                      shared TriggerAuthentication
overlays/
  kind/                      local dev
  eks-staging/               bioagents-staging namespace (dev branch)
  eks-prod/                  bioagents-prod namespace (main branch)
```

Each overlay supplies its own `namespace.yaml`, `config.env` (non-secret env), `secret.env` (LLM keys, Redis URL, Supabase service key), and `pull-secret.env` (GHCR PAT). The `.env` files are generated by CI from the secret store and are gitignored — only `*.env.example` is checked in.

## Apply

```bash
kubectl apply -k k8s/overlays/kind
kubectl apply -k k8s/overlays/eks-staging
kubectl apply -k k8s/overlays/eks-prod
```

Production deploys go through GitHub Actions; only run `kubectl apply` manually for kind or break-glass.

## KEDA prerequisite

Install KEDA into the cluster once:

```bash
kubectl apply --server-side -f https://github.com/kedacore/keda/releases/latest/download/keda-2.x.x-core.yaml
```

(Pin to a specific version in production.)
